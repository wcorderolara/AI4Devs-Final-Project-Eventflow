# 3. Recomendación para métricas del panel admin

Para el MVP, el panel admin no debería convertirse en un dashboard analítico complejo. Debe servir para demostrar gobernanza, actividad de la plataforma y avance del demo.

## 3.1 Métricas recomendadas para MVP

| Métrica                              | Propósito                                       | Prioridad   |
| ------------------------------------ | ----------------------------------------------- | ----------- |
| Total de usuarios registrados        | Mostrar actividad general de la plataforma      | Must Have   |
| Total de organizadores               | Mostrar base de usuarios organizadores          | Must Have   |
| Total de proveedores                 | Mostrar base de vendors                         | Must Have   |
| Proveedores pendientes de aprobación | Apoyar flujo admin de moderación                | Must Have   |
| Proveedores aprobados                | Mostrar catálogo disponible                     | Must Have   |
| Total de eventos creados             | Mostrar uso principal del producto              | Must Have   |
| Eventos por estado                   | Ver `draft`, `active`, `completed`, `cancelled` | Must Have   |
| Total de QuoteRequests creadas       | Medir uso del flujo de cotización               | Must Have   |
| Total de Quotes respondidas          | Medir respuesta de proveedores                  | Must Have   |
| Reviews publicadas                   | Mostrar feedback del sistema                    | Should Have |
| Reviews ocultas/eliminadas           | Mostrar moderación admin                        | Should Have |
| AI recommendations generadas         | Mostrar uso de IA en el producto                | Should Have |
| Eventos completados                  | Mostrar cierre de ciclo de eventos              | Should Have |
| BookingIntents creados               | Mostrar intención de contratación simulada      | Should Have |

## 3.2 Métricas que NO recomiendo para MVP

| Métrica                         | Motivo                               |
| ------------------------------- | ------------------------------------ |
| Ingresos reales                 | No hay pagos reales en MVP           |
| Comisiones reales               | No hay comisión transaccional en MVP |
| CAC / LTV                       | No aplica para demo académica        |
| Conversión comercial real       | No hay operación comercial activa    |
| Revenue por proveedor           | No aplica todavía                    |
| ROI por proveedor               | Futuro, requiere operación real      |
| Métricas avanzadas de marketing | Fuera del alcance del MVP            |

## Decisión recomendada

> El panel admin del MVP debe mostrar métricas de actividad, gobernanza, uso de IA, cotizaciones y demo readiness, no métricas comerciales reales.

# 4. Recomendación para límite de QuoteRequest activas por evento

Aunque actualmente no existe un límite definido, sí recomiendo establecer uno para el MVP.

## Recomendación

```text
Máximo 5 QuoteRequests activas por categoría de servicio por evento.
```

## Justificación

Esto evita que un organizador mande cotizaciones ilimitadas a muchos proveedores para la misma categoría, lo cual podría generar ruido en la demo y complicar el flujo.

Ejemplo:

* Para fotografía: máximo 5 solicitudes activas.
* Para catering: máximo 5 solicitudes activas.
* Para música/DJ: máximo 5 solicitudes activas.

## Regla sugerida

```text
Un evento puede tener múltiples QuoteRequests, pero no más de 5 solicitudes activas por categoría de servicio al mismo tiempo.
```

## Estados que contarían como activas

| Estado      | Cuenta como activa |
| ----------- | -----------------: |
| `sent`      |                 Sí |
| `viewed`    |                 Sí |
| `responded` |                 Sí |
| `preferred` |                 Sí |
| `cancelled` |                 No |
| `expired`   |                 No |
| `rejected`  |                 No |

## Alternativa más simple

Si quieres menos complejidad:

```text
Máximo 20 QuoteRequests activas por evento.
```

Pero mi recomendación es mejor por categoría:

> **5 activas por categoría por evento.**

Esto se siente más realista y evita sobrecargar al proveedor.

# 5. Recomendación sobre AnthropicProvider

## Decisión recomendada para MVP

> Para el MVP, implementar únicamente la interfaz de proveedor IA y OpenAIProvider como proveedor funcional principal. AnthropicProvider puede quedar como implementación futura o stub no operativo.

## Alcance MVP recomendado

| Componente                              | Estado MVP             |
| --------------------------------------- | ---------------------- |
| `AIProvider` interface                  | Incluido               |
| `OpenAIProvider`                        | Incluido               |
| `MockAIProvider`                        | Incluido               |
| `AnthropicProvider`                     | Futuro / Stub opcional |
| Selector dinámico de proveedor IA en UI | No requerido           |
| Comparación real OpenAI vs Anthropic    | No requerida           |
| Failover automático OpenAI → Anthropic  | No requerido           |

## Justificación

Implementar Anthropic funcional desde el MVP agrega complejidad innecesaria:

* configuración adicional,
* manejo de otro SDK/API,
* diferencias de formato,
* pruebas adicionales,
* manejo de errores duplicado,
* potencial aumento de costo,
* más superficie técnica para una demo académica.

La mejor decisión para el MVP es:

```text
AIProvider interface
→ OpenAIProvider funcional
→ MockAIProvider para testing/demo
→ AnthropicProvider preparado como contrato futuro
```

## Regla recomendada

```text
El sistema debe diseñar la integración IA mediante una interfaz de proveedor para evitar acoplamiento directo a un único LLM. En el MVP, OpenAI será el proveedor funcional principal y MockAIProvider será usado para pruebas/demo. AnthropicProvider queda diferido o como stub técnico sin obligación funcional.
```

## 16. ¿Admin puede listar todos los eventos o solo agregados?

## Recomendación

> El admin podrá ver un listado de eventos para fines de demo, soporte y gobernanza, pero con información limitada y sin editar el contenido del evento salvo acciones administrativas permitidas.

## Decisión sugerida

| Capacidad                         | MVP                                                              |
| --------------------------------- | ---------------------------------------------------------------- |
| Ver métricas agregadas de eventos | Sí                                                               |
| Ver listado de eventos            | Sí                                                               |
| Ver detalle completo de evento    | Solo para demo/admin support                                     |
| Editar evento de un organizador   | No                                                               |
| Eliminar evento de un organizador | No, salvo acción administrativa futura                           |
| Cambiar estado de evento          | No, salvo automatización del sistema o acción controlada de demo |

## Regla sugerida

```text
El administrador puede listar eventos y ver su estado general para fines de demo, soporte y gobernanza, pero no puede editar eventos propiedad de organizadores en el MVP.
```

## 17. ¿Admin puede gestionar EventType desde el panel?

## Recomendación

> Sí, pero de forma controlada.

## Decisión sugerida

El admin podrá gestionar `EventType` desde el panel solo para:

* activar/desactivar tipos de evento,
* editar nombre visible,
* editar descripción,
* definir orden de visualización.

No recomiendo permitir eliminar físicamente tipos de evento si ya existen eventos asociados.

## Regla sugerida

```text
El administrador puede gestionar tipos de evento activos para el MVP, pero no puede eliminar físicamente un EventType si ya está asociado a eventos existentes.
```

## Tipos iniciales

* Bodas
* XV años
* Bautizos
* Baby showers
* Cumpleaños
* Eventos corporativos

## 18. ¿Las acciones admin sobre categorías permiten jerarquías?

## Recomendación

> Sí, permitir una jerarquía simple de máximo 2 niveles.

## Ejemplo

```text
Catering
  └── Mesa de dulces

Música
  └── DJ
  └── Marimba
  └── Mariachi

Decoración
  └── Flores
  └── Mobiliario
```

## Decisión sugerida

| Capacidad             | MVP                                         |
| --------------------- | ------------------------------------------- |
| Categorías padre      | Sí                                          |
| Subcategorías         | Sí                                          |
| Más de 2 niveles      | No                                          |
| Reordenamiento manual | Could Have                                  |
| Eliminación física    | No recomendada si tiene servicios asociados |

## Regla sugerida

```text
Las categorías de servicio pueden tener jerarquía simple de hasta 2 niveles en el MVP. No se permiten jerarquías profundas para evitar complejidad innecesaria.
```

## 19. Política de hard vs soft delete para Attachment

## Recomendación

> Usar soft delete lógico para metadata y eliminación controlada del archivo físico.

## Decisión sugerida

Para el MVP:

* Al eliminar una imagen del portafolio, se marca como `deleted` o `inactive`.
* Se mantiene metadata mínima para auditoría.
* El archivo físico puede eliminarse de storage posteriormente mediante proceso técnico.
* No se muestra al público después de eliminarse.

## Regla sugerida

```text
Los attachments eliminados por el usuario o admin deben ocultarse de la interfaz pública mediante soft delete. La eliminación física del archivo podrá realizarse posteriormente por proceso técnico o mantenimiento.
```
